## General Commands

| Command | Description | Options |
| ------- | ----------- | ------- |
| `vm0 info` | Display environment information | - |
| `vm0 init` | Initialize a new VM0 project with `vm0.yaml` | `-f, --force` |

## Authentication

| Command | Description | Notes |
| ------- | ----------- | ----- |
| `vm0 auth login` | Log in to VM0 (opens browser) | Use `VM0_API_URL` env var for custom API |
| `vm0 auth logout` | Log out and remove credentials | - |
| `vm0 auth status` | Show authentication status | - |
| `vm0 auth setup-token` | Output token for CI/CD | - |

## Agent Management

| Command | Description | Options |
| ------- | ----------- | ------- |
| `vm0 compose <config-file>` | Create or update agent compose | `--push` |
| `vm0 run <agent> <prompt>` | Run a task with specified agent | `--vars`, `--secrets`, `--artifact-name` |
| `vm0 run continue <session-id> <prompt>` | Continue from session | Same as `run` |
| `vm0 run resume <checkpoint-id> <prompt>` | Resume from checkpoint | Same as `run` |
| `vm0 cook [prompt]` | Prepare and execute agent from `vm0.yaml` | `-y, --yes` |
| `vm0 agent list` | List all agent composes | `--json` |
| `vm0 agent inspect <name[:version]>` | Inspect agent configuration | `--json` |

### `vm0 run` Examples

```bash
# Run latest version
vm0 run my-agent "build a todo app"

# Run specific version
vm0 run my-agent:abc123 "build a todo app"

# With variables and secrets
vm0 run my-agent "deploy" \
  --vars ENV=production \
  --secrets API_KEY=sk-xxx

# Continue a session
vm0 run continue 9907d243-c38a-442b-84ee-efefd4eaff44 "add authentication"

# Resume from checkpoint
vm0 run resume 61cddb1f-b83a-4b89-8785-97df099fe61f "try different approach"
```

### `vm0 cook` Examples

```bash
# Prepare artifact, compose and run agent based on vm0.yaml and agent.md
vm0 cook

# Same as above with custom prompt
vm0 cook "build a new feature"
```

## Volume Management

Volumes provide persistent file storage that can be shared across agent runs.

| Command | Description | Options |
| ------- | ----------- | ------- |
| `vm0 volume init` | Initialize volume in current directory | `--name <name>` |
| `vm0 volume push` | Push local files to cloud | `--message <msg>`, `--force` |
| `vm0 volume pull [versionId]` | Pull cloud files to local | - |
| `vm0 volume status` | Show volume status and changes | - |
| `vm0 volume list` | List all remote volumes | Alias: `ls` |
| `vm0 volume clone <name> [dest]` | Clone remote volume locally | - |

## Artifact Management

Artifacts are work products generated by agents during execution.

| Command | Description | Options |
| ------- | ----------- | ------- |
| `vm0 artifact init` | Initialize artifact in current directory | `--name <name>` |
| `vm0 artifact push` | Push local files to cloud | `--message <msg>`, `--force` |
| `vm0 artifact pull [versionId]` | Pull cloud artifact to local | - |
| `vm0 artifact status` | Show artifact status and changes | - |
| `vm0 artifact list` | List all remote artifacts | Alias: `ls` |
| `vm0 artifact clone <name> [dest]` | Clone remote artifact locally | - |

## Monitoring

| Command | Description | Options |
| ------- | ----------- | ------- |
| `vm0 logs <runId>` | View logs for an agent run | `-a/--agent`, `-s/--system`, `-m/--metrics`, `-n/--network`, `--since`, `--tail`, `--head` |

**Examples:**
```bash
# View agent events (default)
vm0 logs 11007483-20b2-42b1-8f2a-b42a10186810

# View system logs
vm0 logs 11007483-20b2-42b1-8f2a-b42a10186810 --system

# Show last 20 entries
vm0 logs 11007483-20b2-42b1-8f2a-b42a10186810 --tail 20

# Show logs since 5 minutes ago
vm0 logs 11007483-20b2-42b1-8f2a-b42a10186810 --since 5m
```

## Environment Variables

- `VM0_TOKEN` - Authentication token for API access
- `VM0_API_URL` - Custom API URL (defaults to https://api.vm0.ai)
- `VM0_LOG_LEVEL` - Log level (debug, info, warn, error)

## Frequently Asked Questions

<Accordions className="faq-accordions">

<Accordion title="What's the difference between vm0 cook and vm0 run?">

**`vm0 cook`** is a complete workflow that:
1. Prepares artifacts from your local files
2. Composes the agent using `vm0.yaml` and `agent.md`
3. Executes the agent

Use `-y` flag to skip confirmation prompts.

**`vm0 run`** directly executes an already-composed agent by name:
- Requires specifying the agent name and prompt
- Supports `--vars` and `--secrets` for environment variables
- No artifact preparation or compose step
- Faster for running existing agents

**Use `vm0 cook` when:**
- Developing locally with `vm0.yaml` configuration
- You need to update agent instructions or artifacts
- Starting a new project

**Use `vm0 run` when:**
- Running a published/composed agent
- You need to pass variables or secrets
- Running agents from CI/CD

</Accordion>

<Accordion title="How do I pass environment variables to an agent?">

Use `--vars` for non-sensitive variables and `--secrets` for sensitive data:

```bash
vm0 run my-agent "deploy" \
  --vars ENV=production \
  --vars REGION=us-west \
  --secrets API_KEY=sk-xxx \
  --secrets DB_PASSWORD=pwd123
```

In your agent configuration, reference them as:
- Variables: `${{ vars.ENV }}`
- Secrets: `${{ secrets.API_KEY }}`

**Note:** Secrets are encrypted and not visible in logs.

</Accordion>

<Accordion title="Can I run multiple agents in parallel?">

Yes, you can run multiple `vm0 run` or `vm0 cook` commands in parallel by running them in separate terminal sessions or using background processes:

```bash
# Run in background
vm0 run agent1 "task 1" &
vm0 run agent2 "task 2" &

# Wait for all to complete
wait
```

Each run gets a unique Run ID and executes independently.

</Accordion>

<Accordion title="What's the difference between continue and resume?">

**`vm0 run continue <session-id>`** continues from the latest state:
- Uses the most recent conversation history
- Uses the latest artifact version
- Best for ongoing iterative work

**`vm0 run resume <checkpoint-id>`** restarts from a specific checkpoint:
- Goes back to that exact point in time
- Useful for trying different approaches from the same starting point
- Like a "save point" in a game

</Accordion>
